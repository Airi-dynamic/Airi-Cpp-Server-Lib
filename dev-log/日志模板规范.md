# dev-log 日志模板规范

本文件用于约束后续 dayn 开发日志的统一风格与详尽程度。任何新写或重写的 dev-log，都应满足下方说明为最佳实践。

---

## 1. 文档骨架

每篇 dev-log 顶层结构固定：

```
# Day NN — 标题（用 / 分隔的关键改进）

## 0. 阅读指南
## 1. 引言（这一天为什么要做这些改动）
## 2. 改进 A — <名字>
## 3. 改进 B — <名字>
...
## 9. 工程化相关配置（gitignore / CI / 构建脚本变化等）
## 10. 验证（构建命令，演示观察效果的方式等）
## 11. 局限与下一步（明确说明遗留问题与 day NN+1 计划）
```

每个 "改进 X" 子节进一步固定为：

```
### X.1 业务场景        — 介绍业务场景，问题上下文，可以结合代码和示例逐步追踪的解释
### X.2 改进思路    		— 介绍相关知识，设计思想等
### X.3 编码实现步骤     — 摘抄实现原文，注释中进行必要的解释
### X.4 嵌入执行路径     — 摘抄项目中对这些代码的使用方式，理清“这个改进如何发挥作用”
### X.5 全流程追踪       — 见本规范第 3 节
```

---

## 2. 代码引用规则

1. **任何代码块必须原封不动摘抄自源文件**，不得伪代码、不得改写、不得简化。
2. **每个代码块上方必须以"来自 [显示路径](相对路径)"形式注明出处**。例：
   `来自 [HISTORY/day28/common/Connection.cpp](HISTORY/day28/common/Connection.cpp)：`
3. 若需要省略中间行，用 `...` 单行表示，且不可跨函数。
4. 若同一函数被两处引用，可分两个独立代码块；不要拼接成"伪整段"。
5. 整个文档必须涵盖对应day发生了更改的所有代码，不可遗漏或省略任何一处改动。**判断标准**：与上一日 baseline 做 `git diff` 后产生的每一段代码（新增 / 修改 / 删除），文中都必须有一个对应的代码块被引用并解释；删除的代码亦需以"旧版长这样 → 现在已删除/替换"的形式出现。
6. **逐代码块解释部分必须以"教程"形式组织**——读者按文档从头读到尾，应当能像跟一个 step-by-step tutorial 一样**从 day N-1 的代码状态一步步动手实现到 day N 的代码状态**。具体要求：
   - **顺序**：代码块的出现顺序 = 读者实际动手实现的合理顺序（自底向上：先数据结构 / 类型 → 再纯函数 / 工具 → 再嵌入路径 / 调用方 → 最后测试与构建脚本）；不得按"文件字典序"或"随机思维流"排布。
   - **每个代码块必须配三段说明**：
     (a) **为什么写它**——它解决前一步暴露的什么问题 / 接口缺口；
     (b) **它怎么工作**——逐行 / 逐分支讲清运行逻辑，关键变量代入真实数值；
     (c) **它和上下文怎么衔接**——它依赖前面哪段代码 / 谁会调用它 / 它把哪个状态推进到下一步。
     并注意避免仅仅依赖纯自然语言描述，结合代码块，示例或ASCII图示等进行解释。
   - **不允许跳跃**：
     不得出现"这段代码大家自己看就好"、"细节略"、"实现简单不展开"之类的甩锅式描述；
     任何在 diff 中存在的代码都必须有 (a)(b)(c) 三段。
   - **运行方式**：文中至少要给出一处"读者照着做就能跑通"的命令序列（cmake 配置 / 编译目标 / 启动 demo / 验证输出），不能只贴代码不讲怎么跑起来看效果。

## 3. 有状态类 / 多线程类的「概念先行四步模板」

> **适用范围**：任何新增的、含有内部状态 + 多个互相依赖函数的类（典型：网络客户端、状态机、Actor 节点）。
> 这类类的最常见讲解陷阱：一上来就列函数签名，或者直接进入"这段代码的逻辑是……"，
> 读者没有整体地图，在追踪细节时持续迷失方向。

四步必须**按顺序**出现，且每步必须在代码块之前完成，不可倒置：

---

### 步骤一：场景驱动的问题声明（Why Does This Class Exist？）

用 **3-5 句话**，从读者角度回答"如果没有这个类，我会怎么写，会遇到什么具体问题"。
禁止使用"为了更好地支持 X"之类的虚话。必须是具体的失败场景，例如：

> 旧版 RpcClient::call() 在 EventLoop 线程里执行会阻塞整个 reactor 15 秒（等 connect 超时），
> 导致在此期间所有其他连接无法收发数据。三节点 Raft 集群里，一旦某个 peer 宕机，
> 整个 Leader 就会因此冻结，无法给其余节点发心跳，造成集群无谓的重新选举。

---

### 步骤二：线程模型 + 所有权图

用 ASCII 图明确标出：
- 哪些函数可以在任意线程调用（"外部 API"）
- 哪些函数只能在归属线程执行（"内部实现"）
- 状态成员属于哪条线程

示例格式：

```
外部线程（任意）                loop_ 线程（归属线程）
─────────────────────           ──────────────────────────────
callAsync(...)   ─ runInLoop ──▶  doCall()
                                  ├─ startConnectLocked()
                                  ├─ onResponse()
                                  └─ handleConnectionClosed()
stop()           ─ queueInLoop ─▶  [清理 conn_ / channel_]

         不可变配置（任意线程只读）
         ─────────────────────────
         loop_*, ip_, port_
```

---

### 步骤三：数据成员生命周期表

按"谁拥有 / 何时可改"把所有数据成员分组，每组一行说明。强制三组：

| 分组 | 代表成员 | 生命周期说明 |
|------|---------|-------------|
| 不可变配置（构造后只读） | `loop_` `ip_` `port_` | 任意线程可读，无需加锁 |
| 归属线程独占（仅 loop_ 线程读写） | `state_` `conn_` `pending_` | 所有读写都必须在 loop_ 线程 |
| 原子对外暴露（跨线程快照） | `currentTerm_` `state_`（atomic） | 外部线程只读，归属线程写 |

若某成员不属于这三类，必须在表下方单独注释说明。

---

### 步骤四：函数依赖层次表（Bottom-Up）

从叶函数到根函数自底向上列出，每行标明"被谁调用"。
这张表的作用是让读者在阅读具体实现前，先知道"我现在读的这个函数，是整个调用树的第几层"。

示例格式：

| 层次 | 函数 | 被谁调用 | 一句话职责 |
|------|------|---------|-----------|
| 叶层 | `failAllPending` | stop / startConnect / handleConnectionClosed | 将所有 pending 回调以 ok=false 触发并清空 |
| 叶层 | `cleanupConnectChannel` | onConnectWritable / connect-timer | 注销 connectChannel_ 并关闭 fd |
| 中层 | `startConnectLocked` | doCall（kIdle 时）/ handleConnectionClosed | 发起非阻塞 TCP connect，推进状态到 kConnecting |
| 中层 | `onConnected` | startConnectLocked（立即完成）/ onConnectWritable | 将 raw fd 包装成 Connection，推进状态到 kConnected |
| 根层 | `callAsync` | 外部任意线程 | 跨线程入口；切回 loop_ 线程再调 doCall |

**只有在四步全部完成之后**，才应进入 §X.3 的逐函数代码解释。

---

### 反模式示例（禁止出现）

- ❌ 直接在"改进 X"下面放代码，没有问题声明
- ❌ 用自然语言平铺"函数 A 调函数 B，B 调 C"，不给层次图
- ❌ 数据成员在代码摘录里零散出现，从未做过分组说明
- ❌ 线程所有权含糊（"在某个线程里运行"）
- ❌ 只讲 happy path，不讲 error path 和 timeout path

## 4. 全流程追踪格式（核心）

用来让读者真正"理解"每条新 API 的运行逻辑，而非仅仅"知道"它存在。

### 4.1 必须包含的四块内容

每个 `X.5 全流程追踪` 子节由下面四块按序组成：

#### 第 1 块：业务场景

- 一段"业务场景"描述：用具体示例说清"这条流程在生产中长什么样"。
- 注意描述请勿过于简洁，此段并不限长，请像是在对话中交互式为用户解答疑惑建立实感一样，将此次更改的问题场景上下文厘清。
- 可以结合代码块，示例和ASCII图说明

#### 第 2 块：逐函数代码追踪

按时序总览表中"决策点"为节奏，每个决策点开一个 `#### 第 N 步：xxx` 小标题，内部固定结构：

1. 一句话说明"现在进入了哪个函数 / 哪一段"。
2. **代码块**：必须摘抄该函数对应的源代码上下文，以便读者定位并理解（见第 2 节规则）。
3. **代入实参**：用文字 + 公式形式列出此刻每个参数 / 成员变量的真实值。例：
   
   > 传入实参：`bufferedBytes = 16.06 MB`、`readPaused = false`、`cfg = {4MB, 16MB, 64MB}`。
4. **判断分支走向**：以列表形式逐条复盘 if 条件的 true/false 与最终选择的分支。例：
   
   > - `16.06 MB > 64 MB`？否 → 跳过 close 分支。
   > - `!false && 16.06 MB > 16 MB`？是 → `d.shouldPauseRead = true`。
5. **副作用清单 + 此刻全局状态**：用 fenced code block 列出本步执行后系统进入的快照状态：
   ```
   outputBuffer_              = 16.06 MB
   readPausedByBackpressure_  = true
   epoll 监听位               = EPOLLOUT
   ```
6. **(可选)** 一段"为什么这样设计"的注解，把决策与设计意图绑定。

#### 第 3 块：状态机全景 / ASCII 图（如适用）

若改进涉及多状态切换（如回压、连接生命周期、半包→完整包的状态机），必须画一个 ASCII 状态图，标出阈值、转移条件、副作用动作。
若改进涉及复杂的所有权关系，包含关系等，也需要使用ASCII图说明成员结构等。

#### 第 4 块：各函数职责一句话表

最后用一张表收尾，列出本次涉及的全部函数 / 角色，每个一句话说明"在哪个时机调用 + 它的核心职责"。例：

| 函数 | 函数名语义 | 调用时机 | 职责 |
|------|---------|------|------|
| `evaluateBackpressure` | 回压评估 | applyBackpressure / tryResume 内部 | 纯判断，无副作用 |
| `applyBackpressureAfterAppend` | 追加数据后应用回压决策 | send() 之后 | 暂停读 / 触发兜底断连 |

### 4.2 反面例子（禁止出现）

- ❌ "解出 /a 完整请求 → retrieve(18) → 回调 → reset" 这种**单行高度抽象**的流程描述。
- ❌ 时序表只列动作，不列状态量数值。
- ❌ "完美。" 这种没有信息量的总结。
- ❌ 用伪代码 / 注释片段代替源代码。
- ❌ 只追踪 happy path，忽略 timeout path 和 error path（这两条路径往往才是真正的难点）。
- ❌ 追踪一个有状态类的调用前，没有先完成 §3 的四步模板（读者此时没有地图）。

## 5. 业务场景写作要求

- **MUST 用真实可理解示例作为入口**，例如：
  - "有 3 台机器，它们都存了同一份数据。其中一台挂了，剩下两台怎么保证数据还是一致的？Leader 是谁？谁有权写入？"

## 6. 末尾"局限与下一步"格式

- 列出本日改动的真实遗留（如"回压只覆盖 OutputBuffer 单方向"）。
- 末尾一行用自然语言衔接到下一天："接下来 Day NN+1 会..."。

## 7. 文件物理结构

- 文件采用 LF 换行、UTF-8 无 BOM。

## 8. 自检清单

- [ ] 全流程追踪里**没有**任何"高度抽象一行式描述"的退化。
- [ ] 所有代码块上方都有"来自 [显示路径](路径)"。
- [ ] 末尾"局限与下一步"明确指向下一天。
- [ ] `HISTORY/dayNN/dev-log.md` 与主文件 `diff` 为空。
- [ ] **有状态类自检**：该类是否完成了 §3 的四步模板？（问题声明 → 线程模型图 → 数据成员表 → 函数层次表）。若未完成，说明讲解从代码层面开始，读者没有地图。
- [ ] **教程性自检**：把 day N-1 的代码 checkout 出来，照着 dev-log 顺序逐个粘贴代码块，能否最终重建出 day N 的代码？
  若答"否"，说明顺序错乱 / 代码遗漏 / 解释跳跃。
- [ ] **diff 覆盖自检**：`git diff dayN-1..dayN` 输出的每一段代码（新增/修改/删除），都能在本日 dev-log 里找到对应代码块？
  若答"否"，说明文档不完整。
- [ ] **多路径自检**：含有复杂状态机的改进（连接状态、选举状态等），是否追踪了至少 happy path + timeout/error path 两条路径？
